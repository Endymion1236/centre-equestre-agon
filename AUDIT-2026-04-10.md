## 🔴 CRITIQUES (à corriger immédiatement)

### C1 — `/api/debug-firebase` expose des secrets en production
**Sévérité : CRITIQUE**

Cette route GET publique retourne sans aucune authentification :
- Le `projectId` Firebase
- Les 20 premiers caractères du `clientEmail`
- Les 30 premiers caractères de la `FIREBASE_PRIVATE_KEY`
- Des infos sur le format de la clé privée

**Impact** : Un attaquant peut reconstituer partiellement la clé privée et obtenir des informations facilitant une attaque ciblée.

**Action** : Supprimer cette route immédiatement.

---

### C2 — 19 routes API sans aucune authentification
**Sévérité : CRITIQUE**

Sur 34 routes API, seules 4 vérifient un `verifyIdToken()` et 5 utilisent `CRON_SECRET`. Les 19 suivantes sont totalement ouvertes :

| Route | Risque |
|-------|--------|
| `/api/send-email` | Envoi d'emails arbitraires via votre compte Resend |
| `/api/send-email/test` | Envoi de test + listing des domaines Resend |
| `/api/push` | Envoi de notifications push à toutes les familles (broadcast) |
| `/api/ia` | Consommation illimitée de l'API Anthropic Claude |
| `/api/whisper` | Consommation illimitée de l'API OpenAI Whisper |
| `/api/tts` | Consommation illimitée de l'API ElevenLabs |
| `/api/agent` | Agent IA avec accès écriture Firestore (création créneaux, emails) |
| `/api/cawl/checkout` | Création de sessions de paiement CAWL arbitraires |
| `/api/send-payment-link` | Envoi de liens de paiement par email |
| `/api/invoice-pdf` | Génération de factures PDF |
| `/api/facture` | Génération de factures HTML |
| `/api/avoir-pdf` | Génération d'avoirs PDF |
| `/api/bon-cadeau` | Génération de bons cadeaux |
| `/api/gift-voucher` | Génération de bons cadeaux (doublon ?) |
| `/api/progression-pdf` | Génération de PDF progression |
| `/api/challenges` | Lecture de tous les challenges (données internes) |
| `/api/test-seed` | **Écriture directe Firestore** — injection de données arbitraires |
| `/api/admin/unenroll-annual` | Désinscription de forfaits annuels |
| `/api/admin/assign-ordre-equides` | Modification de l'ordre des équidés |

**Impact** : N'importe qui peut envoyer des emails en votre nom, consommer vos crédits API (Claude, Whisper, ElevenLabs = coûts directs), injecter des données dans Firestore, générer des factures frauduleuses, et envoyer des notifications push.

**Action** : Ajouter `adminAuth.verifyIdToken()` sur toutes les routes admin/sensibles. Pour les routes client, vérifier au minimum que l'utilisateur est authentifié.

### C3 — `/api/test-seed` permet l'injection de données en production
**Sévérité : CRITIQUE**

Aucune vérification d'environnement ni d'authentification. Accepte un POST avec `action: "seed"` et écrit directement dans Firestore via `adminDb`.

**Action** : Supprimer cette route ou la protéger par `PLAYWRIGHT_TEST_MODE` comme `/api/test-auth`.

---

### C4 — Pas de Firestore Security Rules dans le repo
**Sévérité : CRITIQUE**

Aucun fichier `firestore.rules`, `storage.rules` ni `firebase.json` dans le dépôt. Impossible de vérifier que les règles Firestore sont correctement configurées. Si les règles par défaut (mode test) sont encore actives, toutes les données sont lisibles et modifiables par n'importe qui.

**Action** : Exporter les règles depuis la console Firebase, les ajouter au repo, et vérifier qu'elles restreignent correctement l'accès.

---

### C5 — Middleware ne protège que les routes `/api/cron/*`
**Sévérité : CRITIQUE**

Le middleware Next.js ne matche que `/api/cron/:path*`. Les routes `/admin/*` côté client ne sont protégées que par du code client (`useAuth`), ce qui est contournable.

**Action** : Étendre le middleware pour vérifier le token Firebase sur toutes les routes `/api/admin/*` au minimum.

## 🟡 MOYENS (améliorations recommandées)

### M1 — 1103 usages de `any` dans le TypeScript
Réduit considérablement la sûreté de type et peut masquer des bugs.

### M2 — Fichiers monolithiques (4 fichiers > 1000 lignes)
- `EnrollPanel.tsx` : 2018 lignes
- `paiements/page.tsx` : 1804 lignes
- `comptabilite/page.tsx` : 1738 lignes
- `parametres/page.tsx` : 1531 lignes

### M3 — Référence Stripe résiduelle dans `comptabilite/page.tsx`
Ligne 331 : `if (label.includes("STRIPE") || label.includes("STP"))` — logique de matching bancaire qui référence encore Stripe.

### M4 — `dangerouslySetInnerHTML` dans email-templates
L'aperçu des templates email utilise `dangerouslySetInnerHTML` sur du contenu admin. Risque XSS limité (admin seulement) mais présent si un template contient du JS malveillant.

### M5 — Protection `/api/test-auth` contournable
La route bloque en production SAUF si `PLAYWRIGHT_TEST_MODE === "true"`. Si cette variable est définie dans Vercel, la route devient accessible.

### M6 — Double fallback admin (claims + email list)
Le `isAdmin` côté client utilise les custom claims Firebase en priorité, puis fallback sur `ADMIN_EMAILS`. Ce double système complique l'audit et peut créer des incohérences.
